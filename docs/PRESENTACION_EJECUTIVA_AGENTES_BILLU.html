Presentacion Ejecutiva - Plataforma de Agentes Billu

Fecha: 6 de abril de 2026

Documento de apoyo para preparar 3 slides ejecutivos ante direccion.

Mensaje principal: Billu ya cuenta con una plataforma operativa funcional con capacidades de monitoreo, automatizacion y asistentes. La decision ahora no es rehacerla, sino institucionalizarla, fortalecer su seguridad y escalar los componentes con mayor demanda.

Resumen para exponer en 1 minuto

  • Hoy ya existe una base operativa real para monitoreo, seguimiento, alertamiento, categorizacion y atencion de primer nivel.
  • La plataforma actual corre principalmente sobre Google Workspace y permite un time to market muy rapido.
  • Ya existe tambien una base escalable de agente de WhatsApp sobre Node.js para evolucionar atencion y campañas.
  • La brecha principal no es de funcionalidad, sino de institucionalizacion: seguridad, ambientes, secretos, soporte, gobierno de datos y escalamiento tecnico.

Slide 1. Que podemos hacer hoy con los agentes y automatizaciones construidas

Titulo sugerido del slide: Billu ya opera una capa interna de agentes y automatizacion

Mensaje ejecutivo

La plataforma ya resuelve necesidades reales del negocio y del equipo operativo. No es solo un tablero; es una capa de control con automatizaciones y asistentes conectados a fuentes reales.

Capacidades actuales

  • Monitoreo en tiempo real de enrolamiento Billu N2 y N4.
  • Resumen ejecutivo de cuentas, productos, saldos y plasticos.
  • Analisis de cohortes y comportamiento en los primeros 30 dias.
  • Categorizacion automatica de clientes y exportacion de segmentos.
  • Seguimiento de proyectos, tareas y actividades del equipo.
  • Alertas y recordatorios por correo.
  • Asistente operativo BeBot para consultar, crear y actualizar tareas.
  • Carga de audio, transcripcion y generacion de minutas.
  • Base de agente de WhatsApp para orientacion, contencion y handoff.

Valor para direccion

  • Mayor visibilidad operativa.
  • Menor dependencia de consolidaciones manuales.
  • Reaccion mas rapida ante incidencias.
  • Trazabilidad y seguimiento formal del trabajo.
  • Base reusable para nuevos casos de uso.
Frase sugerida para decirla en el comite: "Hoy ya podemos automatizar lectura, seguimiento, alertamiento, segmentacion y atencion de primer nivel; el siguiente paso es volver esta capacidad mas robusta y sostenible."

Slide 2. Como escalamos la plataforma y que debemos fortalecer

Titulo sugerido del slide: Escalar significa industrializar: mas volumen, mas control, menos riesgo

Ruta de escalamiento propuesta

  • Mantener Google Workspace como capa agil para tableros, operacion interna y automatizaciones de negocio.
  • Mover los componentes de mayor volumen, interaccion externa o criticidad a una arquitectura mas robusta.
  • Usar el agente Node.js de WhatsApp como base para evolucionar atencion, campañas, OCR y handoff a humano.
  • Separar ambientes DEV, UAT y PROD con configuracion, hojas y credenciales independientes.
  • Centralizar secretos, tokens y configuraciones sensibles.
  • Incorporar una base de datos administrada para modulos conversacionales y de alta concurrencia.
  • Agregar observabilidad, monitoreo tecnico y modelo formal de soporte.

Controles de seguridad ya existentes

  • Autenticacion por correo Google.
  • Roles ADMIN y OPERATOR.
  • Sesiones con expiracion e inactividad.
  • Politica minima de contraseñas y lockout por intentos.
  • Bitacora de auditoria.
  • Allowlist de correos externos.
  • Validacion de firma para webhook de WhatsApp.

Brechas a cerrar para institucionalizacion

  • La Web App de Apps Script hoy esta expuesta con acceso amplio y debe revisarse.
  • Hay configuraciones y valores sensibles que deben migrarse a un manejo formal de secretos.
  • Persistir solo en Sheets funciona para agilidad, pero limita escalabilidad y concurrencia.
  • Hace falta separar ambientes, responsables, rollback y control de cambios.
  • Se requiere una matriz formal de accesos, propietarios de datos y procedimientos de soporte.
Frase sugerida para decirla en el comite: "La solucion ya funciona; el riesgo principal hoy no es funcional, sino de sostenibilidad, seguridad, gobierno y escalamiento."

Slide 3. Licencias, APIs conectadas, servicios integrados y costo estimado

Titulo sugerido del slide: La operacion base tiene costo bajo; el gasto variable depende del uso de canales e IA

APIs y servicios conectados hoy

  • Google Sheets, Drive, Docs, Gmail y Apps Script.
  • Google Analytics Data API (GA4).
  • Google Speech-to-Text.
  • Google Text-to-Speech.
  • Outlook via webhook para aperturas reales.
  • WhatsApp Cloud API / Meta.
  • OpenAI para OCR / vision en el agente Node cuando se habilite.
  • Brevo como opcion de correo transaccional.

Backup. Preguntas probables y respuestas sugeridas

Que agentes tenemos hoy?

Tenemos agentes y automatizaciones para monitoreo en tiempo real, seguimiento operativo, categorizacion de clientes, minutas desde audio y una base de agente de WhatsApp para orientacion y handoff.

Con que APIs estamos conectados?

Con Google Workspace, GA4, Speech-to-Text, Text-to-Speech, Outlook webhook, WhatsApp Cloud API y OpenAI opcional para OCR/vision.

Que riesgo existe si lo dejamos como esta?

El riesgo principal no es que deje de funcionar funcionalmente, sino que siga dependiendo de configuraciones manuales, sin ambientes formales, sin manejo robusto de secretos y con escalabilidad limitada en algunos modulos.

Que se necesita para institucionalizar?

Tres cosas: gobierno claro, endurecimiento tecnico y un modelo formal de operacion y soporte.

Que decision le estamos pidiendo a direccion?

Patrocinio formal, definicion de dueños, aprobacion para separar ambientes y presupuesto base para licencias, seguridad y escalamiento controlado.
